1. Analisi del Codice Sorgente
Il server Flask analizzato presenta una gestione del contatore utente (count) all'interno della rotta principale /. La logica di incremento è protetta da un controllo che permette l'aggiornamento solo se il contatore è pari a zero.

if request.headers.get('Content-Type') == "application/json" and count == 0:

Criticità individuate:


Assenza di Atomicità: Il controllo del valore (count == 0) e il suo incremento (+= 1) non avvengono in un'unica operazione atomica.

Ritardo Volontario: L'uso di time.sleep(0.1) all'interno di un ciclo crea una finestra temporale di circa 1 secondo in cui il valore nel database rimane invariato (0), pur essendo la richiesta in fase di elaborazione.

2. Strategia di Attacco (Race Condition)
L'attacco si basa sull'invio di richieste multiple simultanee. Sfruttando la latenza introdotta dal server, è possibile far sì che più processi leggano count = 0 contemporaneamente prima che il primo processo termini la sua esecuzione e aggiorni il valore a 1.

Fasi dell'exploit:
Session Management: Creazione di una sessione persistente per mantenere lo stesso user_id tra le richieste.

Multi-threading: Utilizzo di thread paralleli per saturare il server con richieste POST identiche.

Trigger: Invio di un array JSON di 10 elementi per massimizzare il tempo di sleep del server.
